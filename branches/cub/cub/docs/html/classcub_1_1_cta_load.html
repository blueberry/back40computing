Detailed Description

template<typename InputIterator, int CTA_THREADS, int ITEMS_PER_THREAD, CtaLoadPolicy POLICY = CTA_LOAD_DIRECT, PtxLoadModifier MODIFIER = PTX_LOAD_NONE>
class cub::CtaLoad< InputIterator, CTA_THREADS, ITEMS_PER_THREAD, POLICY, MODIFIER >

The CtaLoad type provides operations for reading global tiles of data into the CTA (in blocked arrangement across threads).

.

Overview

CtaLoad can be configured to use one of three alternative algorithms:
  1. cub::CTA_LOAD_DIRECT. Loads consecutive thread-items directly from the input.

  2. cub::CTA_LOAD_VECTORIZE. Attempts to use CUDA's built-in vectorized items as a coalescing optimization. For example, ld.global.v4.s32 will be generated when T = int and ITEMS_PER_THREAD > 4.

  3. cub::CTA_LOAD_TRANSPOSE. Loads striped inputs as a coalescing optimization and then transposes them through shared memory into the desired blocks of thread-consecutive items
The data movement operations exposed by this type assume a blocked arrangement of data amongst threads, i.e., an n-element list (or tile) that is partitioned evenly among CTA_THREADS threads, with threadi owning the ith segment of consecutive elements.
Template Parameters
InputIteratorThe input iterator type (may be a simple pointer).
CTA_THREADSThe CTA size in threads.
ITEMS_PER_THREADThe number of consecutive items partitioned onto each thread.
POLICY[optional] cub::CtaLoadPolicy tuning policy enumeration. Default = cub::CTA_LOAD_DIRECT.
MODIFIER[optional] cub::PtxLoadModifier cache modifier. Default = cub::PTX_LOAD_NONE.

Important Features and Considerations

  • After any operation, a subsequent CTA barrier (__syncthreads()) is required if the supplied CtaLoad::SmemStorage is to be reused/repurposed by the CTA.
  • The following conditions will prevent vectorization and loading will fall back to cub::CTA_LOAD_DIRECT:
    • ITEMS_PER_THREAD is odd
    • The InputIterator is not a simple pointer type
    • The input offset (ptr + cta_offset) is not quad-aligned
    • The data type T is not a built-in primitive or CUDA vector type (e.g., short, int2, double, float2, etc.)

Examples

  • Example 1: Have a 128-thread CTA load four consecutive integers per thread (blocked arrangement):
    #include <cub.cuh>
    __global__ void SomeKernel(int *d_in, ...)
    {
    // Parameterize a CtaLoad type for use in the current problem context
    typedef cub::CtaLoad<int, CTA_THREADS, 4> CtaLoad;
    // Declare shared memory for CtaLoad
    __shared__ typename CtaLoad::SmemStorage smem_storage;
    // A segment of consecutive input items per thread
    int data[4];
    // Load a tile of data at this block's offset
    CtaLoad::Load(data, d_in, blockIdx.x * 128 * 4);
  • Example 2: Have a CTA load consecutive integers per thread (blocked arrangement) using vectorized loads and global-only caching:
    #include <cub.cuh>
    template <int CTA_THREADS, int ITEMS_PER_THREAD>
    __global__ void SomeKernel(int *d_in, ...)
    {
    // Parameterize a CtaLoad type for use in the current problem context
    typedef cub::CtaLoad<int, CTA_THREADS, ITEMS_PER_THREAD, CTA_LOAD_VECTORIZE, PTX_LOAD_CG> CtaLoad;
    // Declare shared memory for CtaLoad
    __shared__ typename CtaLoad::SmemStorage smem_storage;
    // A segment of consecutive input items per thread
    int data[ITEMS_PER_THREAD];
    // Load a tile of data at this block's offset
    CtaLoad::Load(data, d_in, blockIdx.x * CTA_THREADS * ITEMS_PER_THREAD);
